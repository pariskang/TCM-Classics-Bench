TCM-Classics-Bench builds an evaluation benchmark for large language models on
**classical Chinese-medicine (中醫古籍) texts**. Every question is generated by
"reverse-engineering" from the source passage and is **grounded back to the
original text** — the corpus is the [中醫笈成 / Jicheng-TCM](https://jicheng.tw/tcm/)
free library of punctuated TCM classics (snapshot `book-20180111`, 803 books).

## Why source-grounded?

Classical-text benchmarks fail in two ways: the model answers from modern
textbook knowledge instead of the cited text, or the citation is real but the
*edition* is unknown. This builder fixes both:

- The generator may use **only** the supplied 原文 + metadata; it may not
  invent formulae, herbs, doses, or disease names (enforced in `validate.py`).
- Every item records its `base_text` (底本/參本), `heading_path`, and
  `version_commit`, so each answer is traceable to an exact edition & locus.
- Items needing outside knowledge are tagged `inference_level=external_required`
  and **excluded from the formal test set**.

## Quickstart

```bash
pip install -r requirements.txt

# 1. Download + extract the Jicheng corpus snapshot (~69 MB) -> corpus_src/book/
bash scripts/download_corpus.sh

# 2a. 简易模式 — one command, ~5000 validated questions in seconds, no API key
python -m tcm_bench simple --root corpus_src/book --n 5000 --out test_5k.jsonl

# 2b. Or regenerate all committed artifacts (catalog, corpus, bench items)
python scripts/build_release.py --root corpus_src/book

# --- or drive the pipeline stage by stage ---
python -m tcm_bench catalog  --root corpus_src/book --out data/catalog.jsonl
python -m tcm_bench ingest   --root corpus_src/book --out data/pilot
python -m tcm_bench generate --corpus data/pilot/pilot2_zhongjing.jsonl \
                             --out items.jsonl --tasks T1 T6
python -m tcm_bench validate --items items.jsonl \
                             --corpus data/pilot/pilot2_zhongjing.jsonl
```

The committed `data/bench/*.jsonl` can be re-validated against the committed
`data/corpus/*.jsonl` with no re-ingest, because the corpus sample is a
superset of every cited passage.

### 简易模式 (simple mode)

`simple` is a turnkey path to a balanced, validated test set. It ingests the
pilot corpora, generates items, validates every one against its source, and
spreads the result round-robin across books and tasks:

```bash
# Deterministic (T1 + T6), no API key — 5000 questions, 16 books, ~9 s
python -m tcm_bench simple --root corpus_src/book --n 5000 --out test_5k.jsonl

# Mix in LLM-backed tasks with 16-way concurrency + a progress bar
python -m tcm_bench simple --root corpus_src/book --n 500 --tasks T1 T6 T2 T8 \
                           --llm --provider poe --model Claude-Sonnet-4 \
                           --workers 16 --progress
```

A committed 500-item demo lives at `data/simple_sample.jsonl`.

**Concurrency, progress, real-time writes.** `--workers N` fans the LLM calls
out across a **bounded** thread pool (`generate_items_concurrent`, ~`2N` jobs in
flight) so stopping at the target wastes few calls; `--progress` shows a `tqdm`
bar; both `simple` and `generate` stream each validated item to the output file
as it is produced, so a long run persists incrementally.

**Resume (断点续跑).** `generate_items_concurrent(..., skip=done)` skips a set of
`(passage_id, task_code)` already generated. The Colab notebook uses this to
**continue from where a previous run stopped**: it reads the questions already
written to Google Drive, skips them, and appends in `"a"` mode — so raising the
target (e.g. 5000 → 33000) continues rather than restarting or overwriting. Each
question is flushed to Drive as it is generated, so a dropped session loses
nothing.

### Harder LLM-generated items

LLM tasks are steered toward **reasoning-heavy** questions, not surface
lookups. `--difficulty {Medium,Hard,Expert}` (default **Hard**) injects a
directive that demands multi-step inference, and the reasoning tasks
**T7–T9, T11, T12** are emitted as **4-option single-choice** items with
hard, source-grounded distractors — each distractor must carry an
`exclusion_reason`, and one that needs outside knowledge downgrades the item.
Everything still has to pass source-grounded validation (evidence spans,
formula/herb/dose in source, no `external_required` in the test set).

> **Note (important):** `--llm` only does anything if `--tasks` contains an
> LLM task. `T1` and `T6` are *deterministic* — `--tasks T1 T6 --llm` makes no
> API call (the CLI now warns). For real LLM generation use, e.g.,
> `--tasks T2 T3 T8 T9 T11`.

### LLM providers

LLM-backed tasks (T2–T12) work through a provider-agnostic client
(`tcm_bench/llm.py`). Pick one with `--provider`:

| `--provider` | SDK | Env vars | `--model` example |
| --- | --- | --- | --- |
| `anthropic` (default) | `anthropic` | `ANTHROPIC_API_KEY` | `claude-opus-4-8` |
| `azure` | `openai` | `AZURE_OPENAI_API_KEY`, `AZURE_OPENAI_ENDPOINT`, `AZURE_OPENAI_API_VERSION` | deployment name |
| `poe` | `openai` | `POE_API_KEY` | `Claude-Sonnet-4` |
| `litellm` | `litellm` | provider-specific | `azure/gpt-4o`, `gemini/gemini-1.5-pro` |

```bash
export AZURE_OPENAI_API_KEY=... AZURE_OPENAI_ENDPOINT=https://<res>.openai.azure.com
python -m tcm_bench generate --corpus data/pilot/pilot2_zhongjing.jsonl \
       --out items.jsonl --tasks T2 T3 T8 T9 --llm --difficulty Expert \
       --provider azure --model my-gpt4o-deploy --workers 8 --progress
```

```json
  "question": "請從以下古籍方劑條文中解析方名、組成與劑量。",
  "answer": {
    "formula_name": "桂枝加厚朴杏子湯",
    "ingredients": [
      {"herb": "桂枝", "dose": "三兩", "preparation": "去皮"},
      {"herb": "厚朴", "dose": "二兩", "preparation": "炙，去皮"},
      {"herb": "杏仁", "dose": "五十枚", "preparation": "去皮尖"}
    ]
  },

  },

  "generator": "deterministic"

```
